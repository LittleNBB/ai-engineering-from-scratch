# 并行/群体/网络化架构（Parallel / Swarm / Networked Architectures）

> 与监督者的对比：没有中央决策者。智能体读取共享事件总线，异步拾取工作，将结果写回。LangGraph 明确支持"群体架构"用于去中心化、动态环境。Matrix（arXiv:2511.21686）将控制流和数据流都表示为通过分布式队列传递的序列化消息，以消除编排器瓶颈。权衡是明确的：用确定性和可追溯性换取可扩展性。群体适合有许多独立子问题的任务；不适合需要单一连贯计划的任务。

**类型：** Learn + Build（学习 + 构建）
**语言：** Python（标准库，`threading`，`queue`）
**前置课程：** Phase 16 · 05（Supervisor Pattern），Phase 16 · 04（Primitive Model）
**时间：** ~75 分钟

## 问题

监督者可以扩展到少数工人。那几百个呢？监督者本身成为瓶颈：关于谁做什么的每个决策都通过一个智能体。一个缓慢的规划步骤就会阻塞整个系统。

群体架构翻转了设计。工人从共享队列中拾取工作，而非中央规划者分配工作。"协调"被烘焙到事件总线语义中。没有编排器；系统一直扩展到队列的极限。

## 核心概念

### 形态

```
                ┌──── 共享队列 ────┐
                │                  │
       ┌────────┼────────┐  ◄────┬──┘
       ▼        ▼        ▼       │
     工人 A   工人 B   工人 C   工人 D
       │        │        │       │
       └────────┴────────┴───────┘
                 │
                 ▼
             结果池
```

没有编排器。每个工人重复：拉取任务、处理、写入结果（并可选择地入队后续任务）。

### 群体适用场景

- **许多独立任务。** 抓取、转换、分类。任务之间不相互依赖。
- **可变时长工作。** 如果某些任务花 100 毫秒而其他花 10 秒，群体会自动平衡负载——快速工人拉取下一个任务。监督者必须预估时长。
- **吞吐量优先于确定性。** 你关心总完成时间，而非严格排序。

### 群体失败场景

- **有序工作流。** 如果步骤 3 需要步骤 2 的输出，群体有步骤 3 在步骤 2 完成之前就触发的风险。
- **全局计划任务。** 复杂研究问题受益于规划者。一群研究员产出独立的事实，而非连贯的报告。
- **调试。** 没有中央日志且工作异步，重现 bug 代价高昂。

### Matrix（arXiv:2511.21686）

Matrix 是 2025 年将群体推向自然结论的论文：控制流和数据流都是分布式队列上的序列化消息。没有中央协调器。容错来自消息持久性。可扩展性是消息代理的问题，而非系统的。

贡献：一个编程模型，其中多智能体协调是"这个智能体订阅什么消息主题？"而非"监督者下一个选哪个智能体？"这使系统看起来像发布/订阅事件网格。

### LangGraph 的群体架构

LangGraph 2025 文档明确将"群体架构"描述为多智能体模式之一：智能体是节点，但边形成带有循环的有向图，任何节点都可以从池中被激活。工人按条件从可用工作中拾取，而非由监督者分配。

### 失败模式：饥饿和热点

如果所有工人都拉取最快可用的任务，长时间运行的任务永远不会被拾取，直到它们是唯一剩下的。经典的队列饥饿。

缓解措施：
- 带有显式老化的优先队列（随等待时间增加优先级）。
- 工人专业化：某些工人只接受"长"任务。
- 背压（Back-pressure）：限制快速任务进入队列的数量。

### 与基于内容的路由的关联

群体与基于内容的路由天然配对（Lesson 22）。不是通用队列，而是每种消息类型一个队列。专业化工人只订阅他们的类型。这是扩展到数千个智能体的消息总线架构的基础。

## 动手实现

`code/main.py` 实现了一个由 4 个工人线程组成的群体，从共享的 `queue.Queue` 中拉取。任务具有可变时长（有些快，有些慢）。演示对比了：

- **串行基线：** 一个工人串行处理所有任务。
- **固定分配：** 每个任务预先分配给特定工人（监督者风格）。
- **群体：** 工人从共享队列中拉取。

群体自动平衡负载；固定分配在分配的任务慢时让快速工人闲置。

运行：

```
python3 code/main.py
```

输出展示了每个工人的任务计数（群体不均匀但最优地分配）和挂钟时间。

## 实际应用

`outputs/skill-swarm-fit.md` 评估任务应使用群体还是监督者。输入：任务独立性、时长方差、排序需求、可调试性需求。

## 交付建议

检查清单：

- **带老化的优先队列。** 防止长任务饥饿。
- **工人幂等性。** 如果工人在运行中途崩溃，任务可能被拉取多次。工人必须是幂等的。
- **持久化队列。** 生产中使用 Kafka、Redis Streams 或数据库支持的队列。`queue.Queue` 仅在内存中。
- **每个任务的可观察性。** 每个任务有追踪 ID；每个工人用它记录开始/结束。
- **背压。** 如果队列增长快于工人排空速度，减慢生产者。

## 练习

1. 运行 `code/main.py`。在可变时长工作负载上，群体比串行快多少？比固定分配快多少？
2. 添加优先队列变体（使用 `queue.PriorityQueue`）。按任务"重要性"字段分配优先级。观察低优先级任务在持续负载下是否饥饿。
3. 实现热点检测器：当任何工人处理的任务数是最慢工人的 3 倍时记录日志。这对任务时长分布说明了什么？
4. 阅读 Matrix 论文（arXiv:2511.21686）摘要和第 3 节。识别 Matrix 接受的一个具体权衡（可扩展性收益）和它放弃的一个（可追溯性、确定性）。
5. 将群体演示转换为使用 (task_type, payload) 元组的 `queue.Queue`，工人只订阅特定类型。当任务异构时，什么路由规则合理？

## 关键术语

| 术语 | 常见说法 | 实际含义 |
|------|---------|---------|
| Swarm architecture（群体架构） | "去中心化智能体" | 工人从共享队列中拉取；没有中央编排器。 |
| Event bus（事件总线） | "智能体订阅主题" | 按类型或内容将任务路由到工人的消息代理。 |
| Starvation（饥饿） | "任务永远不运行" | 低优先级任务因高优先级工作持续到达而永远不被拾取。 |
| Hot-spotting（热点） | "一个工人被淹没了" | 一个工人获得大多数任务的负载不均衡。 |
| Back-pressure（背压） | "减慢生产者" | 队列满时信号通知上游停止生产的机制。 |
| Idempotent worker（幂等工人） | "安全重跑" | 处理两次的任务产生相同结果。因工人可能在运行中途崩溃而必需。 |
| Durable queue（持久化队列） | "能撑过崩溃" | 由磁盘或复制存储支持的队列；工人崩溃时任务不丢失。 |
| Matrix framework（Matrix 框架） | "完全消息传递群体" | 数据流和控制流都是分布式队列上的序列化消息。 |
