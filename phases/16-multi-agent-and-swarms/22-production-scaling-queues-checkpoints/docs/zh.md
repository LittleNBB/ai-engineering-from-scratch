# 生产扩展 — 队列、检查点、持久化（Production Scaling — Queues, Checkpoints, Durability）

> 将多智能体系统扩展到数千个并发运行需要**持久化执行（durable execution）**。LangGraph 的运行时在每个超级步骤后按 `thread_id` 写入检查点（默认使用 Postgres）；工作者崩溃后释放租约，另一个工作者恢复。智能体可以无限期地休眠等待人类输入。**MegaAgent**（arXiv:2408.09955）运行每个智能体的生产者-消费者队列，有三种状态（空闲/处理中/响应）和两层协调（组内聊天 + 组间管理聊天）。**Fiber/async** 在 LLM 流式传输上优于线程-每任务：线程 99% 的时间空闲等待 token，fiber 在 I/O 上协作让出。反论：Ashpreet Bedi 的"Scaling Agentic Software"主张在负载证明需要之前只用 **FastAPI + Postgres + 别无其他**——简单架构比预期走得更远。本课构建一个持久化检查点日志、一个带状态转换的每个智能体工作队列、一个 async vs 线程的演示，并落地务实的"从简单开始"规则。

**类型：** Learn + Build（学习 + 构建）
**语言：** Python（标准库，`asyncio`，`sqlite3`）
**前置课程：** Phase 16 · 09（Parallel Swarm Networks），Phase 16 · 13（Shared Memory）
**时间：** ~75 分钟

## 问题

一个原型多智能体系统在一台笔记本上用三个智能体在内存事件循环中运行。你转向生产：

- 智能体有时运行数小时（长时间研究、人在回路等待）。
- 工作者进程崩溃。重启丢失状态。
- 峰值负载是平均的 10 倍；你需要水平扩展。
- 用户按智能体运行付费；你需要精确一次语义来计费。

内存事件循环做不到这些。你需要一个持久化执行层。2026 年的经典选项是：

1. 带检查点的工作流引擎（Temporal、LangGraph 运行时）。
2. 带状态存储的消息队列（Postgres + SQS/RabbitMQ）。
3. Actor 模型框架（MegaAgent 的每个智能体生产者-消费者）。
4. 手工 FastAPI + Postgres（Bedi 的主张）。

本课构建每种的微型版本。

## 核心概念

### 持久化执行，该模式

持久化执行引擎在每个"步骤"（LangGraph 术语中的超级步骤）后持久化完整程序状态。崩溃时：

```
工作者在步骤中途崩溃
  -> 租约超时
  -> 另一个工作者拾取 thread_id
  -> 从最后检查点恢复
  -> 没有重复副作用
```

使其工作的需求：

- **可序列化状态。** 所有智能体状态必须可持久化。带有活跃数据库连接的函数闭包不能存活。
- **确定性恢复。** 给定相同状态和相同输入，智能体产生相同动作（或对外部 LLM 调用委托给外部确定性预言机）。
- **幂等副作用。** 外部调用（工具调用、支付）必须是幂等的或使用去重键。

LangGraph 在每个超级步骤后写入检查点；Temporal 在每个活动后写入；Restate 使用事件源日志。三者都实现了相同模式。

### LangGraph 的运行时

每个智能体有一个 `thread_id`；状态是类型化字典；每个超级步骤向检查点表写入一行。恢复时，运行时从最后检查点重放，而非从头开始。智能体可以 `interrupt()` 等待人类输入；运行时持久化并释放工作者。当输入到达时，任何工作者都可以恢复。

这是 2026 年 4 月的参考生产设计。

### MegaAgent 的每个智能体队列

arXiv:2408.09955 描述了一个规模实验：一个集群中数千个并发智能体。架构：

```
智能体 i:
  state ∈ {Idle, Processing, Response}
  in_queue   <- 发给智能体 i 的消息
  out_queue  -> 回复 + 副作用

协调者:
  组内聊天  （同一组中的智能体）
  组间管理聊天  （高层路由）
```

两层协调使组内对话密集进行而组间保持稀疏——这是保持数千智能体成本线性的模式。

### Async vs 线程-每任务

LLM 调用是 I/O 密集的。等待下一个 token 的线程 99% 的时间空闲。每个线程成本约 1MB 内存；在 10,000 个并发调用时，仅栈就需要 10GB。

Fiber（Python `asyncio`、Go goroutine、Rust `tokio`）在 I/O 上协作让出。同样的 10,000 次调用可以舒适地放入一个进程中。在 LLM 智能体规模上，async 不是优化——它是架构。

例外：CPU 密集的后处理（嵌入、分词器技巧）仍然需要线程或进程。将 I/O 层与 CPU 层分离。

### Bedi 的反论

"Scaling Agentic Software"（Ashpreet Bedi，2026）认为大多数团队在测量负载之前过度工程化。务实的默认：

- FastAPI + Postgres。
- 每次智能体运行是一行；状态通过乐观并发就地更新。
- 通过 `pg_notify` 或简单 Celery 工作者进行后台任务。
- 应用代码中的重试策略。

对于可管理任务上约 100 个并发智能体运行以下的负载，这通常就是你需要的。当测量到失败时再升级。

规则：当你遇到简单架构无法解决的具体问题时，才采用持久化执行框架。过早采用会在不产生回报的仪式上浪费时间。

### 精确一次语义

对于付费智能体运行，你需要"精确一次有效"（至少一次传递 + 幂等消费者）。工程措施：

- **每次运行的去重键。** 在每个副作用调用中包含它。
- **发件箱模式（Outbox pattern）。** 副作用先写入一个表，然后单独的进程执行它们。两步都是幂等的。
- **补偿事务。** 当副作用成功但其跟踪写入失败时，安排补偿。

这些是数据库工程模式，不是 LLM 特定的。LLM 税只是 LLM 调用慢；其他一切都是标准分布式系统。

### 彩虹部署（Rainbow Deployment）

Anthropic 的多智能体研究系统使用"彩虹部署"：智能体运行时的多个版本并发运行，这样长时间运行的智能体不必在每次代码部署时被终止。在流量的一小部分上金丝雀测试新版本；当旧版本的智能体完成后退役。

这对长时间运行的有状态系统是标准做法；2026 年的适应是智能体可以存活数小时，所以部署周期必须配合。

### 经典生产检查清单

- 持久化状态（检查点、快照或发件箱 + 可重放日志）。
- 幂等副作用。
- LLM 调用的异步 I/O 层。
- 至少一次传递加去重。
- 有状态工作负载的彩虹/金丝雀部署。
- 可观察性：每个智能体的跟踪、超级步骤审计、重试计数器。

## 动手实现

`code/main.py` 实现了：

- `CheckpointStore` — SQLite 支持的检查点日志，按 thread-id 键。每个超级步骤追加一行。
- `run_with_checkpoint(agent, thread_id)` — 模拟运行中途崩溃；第二个工作者从最后检查点恢复。
- `AgentQueue` — 每个智能体的空闲/处理中/响应状态机，带小型工作队列。
- `demo_async_vs_threads()` — 通过 asyncio 和线程运行 500 个并发模拟"LLM 调用"；报告挂钟时间和峰值内存（近似）。

运行：

```
python3 code/main.py
```

预期输出：模拟崩溃后检查点恢复成功；async 版本在 < 1 秒内处理 500 个并发调用；线程版本耗时数秒，每个并发单元使用数量级更多的内存。

## 交付建议

经典生产加固：

- **从简单开始（Bedi 规则）。** FastAPI + Postgres，直到测量到失败。
- **在优化之前先监控一切。** 每次运行的延迟直方图、每步时间、重试计数、失败分类。
- **副作用的发件箱模式。** 特别是支付和外部 API 调用。
- **彩虹部署。** 永不在部署期间终止进行中的智能体运行。
- **当**你遇到具体问题时采用持久化执行引擎（Temporal / LangGraph / Restate）：数小时的人在回路等待、跨区域协调、复杂的重试/补偿策略。
- **I/O 层使用 async。** 线程仅用于 CPU 密集的后处理。

## 练习

1. 运行 `code/main.py`。确认检查点恢复有效；测量 async vs 线程并发差异。
2. 实现一个**发件箱**表：每次工具调用先写入发件箱，然后单独的 goroutine/task 执行。通过运行工具调用两次来验证幂等性。
3. 模拟**彩虹部署**：两个并发运行时版本；将一半新 thread_id 路由到每个；确认旧版本上的进行中线程不被中断。
4. 阅读 LangGraph 运行时文档（下方链接）。识别运行时的哪些功能在手工 FastAPI + Postgres 版本中需要最长时间来复制。这是采用的理由，还是你可以推迟？
5. 阅读 MegaAgent（arXiv:2408.09955）第 3 节。两层协调（组内 + 组间管理聊天）是显式的。草拟你如何将其映射到具有两个队列族的消息队列。

## 关键术语

| 术语 | 常见说法 | 实际含义 |
|------|---------|---------|
| Durable execution（持久化执行） | "持久化程序状态" | 引擎在每个超级步骤后写入状态；崩溃恢复是确定性的。 |
| Super-step（超级步骤） | "事务边界" | 检查点之间的工作单元。LangGraph 术语。 |
| thread_id | "智能体运行标识符" | 绑定检查点和恢复逻辑的键。 |
| Idempotency（幂等性） | "安全重试" | 重复副作用产生与一次尝试相同的结果。 |
| Outbox pattern（发件箱模式） | "解耦副作用" | 将意图写入表；单独的执行器执行并标记完成。 |
| At-least-once delivery（至少一次传递） | "可能有重复" | 消息队列语义；去重键使消费者有效一次。 |
| Rainbow deploy（彩虹部署） | "重叠版本" | 长时间运行工作负载期间多个运行时版本并发。 |
| Async fiber（异步 fiber） | "协作让出" | 用户模式并发；对 I/O 密集负载比线程便宜。 |
| Checkpoint（检查点） | "状态快照" | 超级步骤边界处的序列化状态；恢复的关键。 |
