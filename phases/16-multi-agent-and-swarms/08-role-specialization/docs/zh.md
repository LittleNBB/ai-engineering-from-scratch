# 角色专业化 —— 规划者、批评者、执行者、验证者（Role Specialization — Planner, Critic, Executor, Verifier）

> 2026 年最常见的多智能体分解：一个智能体规划，一个执行，一个批评或验证。MetaGPT（arXiv:2308.00352）将其形式化为编码到角色提示词中的 SOP——产品经理、架构师、项目经理、工程师、QA 工程师——遵循 `Code = SOP(Team)` 公式。ChatDev（arXiv:2307.07924）通过"聊天链"将设计师、程序员、审阅者、测试者串联起来，带有"沟通式去幻觉"（智能体明确请求缺失的细节）。验证者是承重角色：Cemri 等人（MAST，arXiv:2503.13657）表明每个多智能体失败都可以追溯到缺失或损坏的验证。PwC 报告称，在 CrewAI 中添加结构化验证循环带来了 7 倍准确率提升（10% → 70%）。

**类型：** Learn + Build（学习 + 构建）
**语言：** Python（标准库）
**前置课程：** Phase 16 · 04（Primitive Model），Phase 16 · 05（Supervisor）
**时间：** ~60 分钟

## 问题

通用多智能体系统产生通用输出。群聊中的三个编码者写出三种口味的平庸代码。你可以添加更多智能体、更多轮次，仍然无法跨越质量阈值。

修复方法不是更多智能体——而是*不同的*智能体。分配不同的角色。给批评者规划者没有的工具。给验证者一个客观的测试套件。现在系统有了基于有据可依的纠正的内部分歧，而不仅仅是并行猜测。

## 核心概念

### 四个经典角色

**Planner（规划者）。** 读取目标，产出步骤列表或规格说明。工具：知识检索、文档。输出：结构化计划。

**Executor（执行者）。** 每次读取一个计划步骤，产出工件。工具：实际的工作工具（代码编译器、shell、API 客户端）。输出：工件。

**Critic（批评者）。** 根据规划者的意图读取执行者的输出。工具：对工件的只读访问、静态分析。输出：接受/拒绝及原因。

**Verifier（验证者）。** 读取工件并运行确定性检查。工具：测试运行器、类型检查器、Schema 验证器。输出：通过/失败及证据。

批评者是主观的、有主见的，通常是基于 LLM 的。验证者是客观的、确定性的，通常是基于代码的。它们不是同一个角色。

### MetaGPT 的 SOP 模式

MetaGPT（arXiv:2308.00352）将软件工程 SOP 编码为角色提示词：

- **产品经理** 编写 PRD。
- **架构师** 产出系统设计。
- **项目经理** 拆分任务。
- **工程师** 实现。
- **QA 工程师** 运行测试。

每个角色都有严格的输入/输出 Schema。角色提示词说明该角色*是什么*以及它*必须产出什么*。`Code = SOP(Team)` 公式——确定性 SOP 将 LLM 团队变成可预测的流水线。

### ChatDev 的沟通式去幻觉

ChatDev 添加了一个关键举措：当执行者需要计划中没有的具体细节时，它在继续之前明确向设计师询问。这防止了 LLM 以看似合理的方式编造细节的经典失败。

实现：角色提示词包含"当你需要未被提供的具体信息时，在产出输出之前按名称询问相关角色"。

### 为什么验证者最重要

Cemri 等人（MAST）追踪了 1642 次多智能体执行失败。21.3% 是验证缺口——系统发布了无人检查的答案。其余 79% 通常可以追溯到"有一个检查静默失败或从未运行"。验证是承重角色。

PwC 报告称（CrewAI 部署，2025），添加结构化验证循环将准确率从 10% 提升到 70%。一个角色带来 7 倍增益。

### 批评者 vs 验证者

- 批评者是一个审查工件质量的 LLM。主观的。可能被看似合理的文字所迷惑。
- 验证者是一个在工件上运行的确定性程序。客观的。给出通过/失败及证据。

两者都要用。批评者捕捉验证者无法表达的品味问题。验证者捕捉批评者看不到的 bug，因为它们只在运行时出现。

### 反模式

你系统中的每个角色都是 LLM，每个角色的输出都是"看起来不错"。经典的 MAST 失败模式。至少添加一个验证者，其通过/失败由代码而非 LLM 决定。

### 框架映射

- **CrewAI** — `Agent(role, goal, backstory)` 是教科书式的专业化表面。
- **LangGraph** — 节点可以有专业化的提示词；边强制执行流水线。
- **AutoGen** — GroupChat 中带有简短名称的角色特定 ConversableAgent。
- **OpenAI Agents SDK** — 角色专业化 Agent 之间的移交工具。

## 动手实现

`code/main.py` 实现了一个 4 角色流水线来构建一个简单的 Python 函数：

- **Planner** 产出规格说明。
- **Executor** 生成代码字符串。
- **Critic**（LLM 模拟）标记明显问题。
- **Verifier** 在沙箱（`exec`）中对生成的代码运行测试用例。

演示运行两次：一次是执行者产出正确代码（批评者 + 验证者都通过），一次是执行者产出偏离规格的代码（批评者因为看起来合理而错过 bug，验证者因为测试失败而捕获它）。

运行：

```
python3 code/main.py
```

## 实际应用

`outputs/skill-role-designer.md` 接受任务并产出角色名册（3-5 个角色）、每个角色的输入/输出 Schema 和验证者检查。在将智能体接入框架之前使用它。

## 交付建议

检查清单：

- **至少一个确定性验证者。** 绝不全是 LLM。
- **每个角色有显式的 I/O Schema。** 规划者返回规格说明，而非散文；执行者读取该 Schema。
- **沟通式去幻觉。** 执行者必须在信息缺失时询问规划者；绝不编造。
- **批评者/验证者排序。** 先运行批评者（便宜，捕捉设计问题），再运行验证者（慢，捕捉 bug）。
- **循环预算。** 最多 2 轮批评者-执行者修订，然后升级到人类。

## 练习

1. 运行 `code/main.py` 并观察验证者如何捕获批评者错过的 bug。添加一个静态分析检查（计数 `return` 出现次数）作为额外验证者。它能捕获运行时测试遗漏的什么？
2. 添加第 5 个角色："需求分析师"，将用户愿望转化为规划者可用的规格说明。哪些沟通式去幻觉请求应该流向它？
3. 阅读 MetaGPT 第 3 节（"Agents"）。列出 MetaGPT 5 个角色各自的输入/输出 Schema。
4. 阅读 ChatDev 的聊天链图（arXiv:2307.07924 Figure 3）。识别沟通式去幻觉在哪里打破了否则会无限循环的流程。
5. PwC 的 7 倍准确率提升来自验证循环。假设三个添加验证者不会有帮助的任务——在这些任务中，正确性的确定性检查是不可能的或代价过高。

## 关键术语

| 术语 | 常见说法 | 实际含义 |
|------|---------|---------|
| Role specialization（角色专业化） | "不同智能体，不同工作" | 为规划者/执行者/批评者/验证者角色调优的不同系统提示词。 |
| SOP 模式 | "编码的标准操作程序" | MetaGPT 的框架：每个角色的严格 I/O Schema 将团队变成流水线。 |
| Communicative dehallucination（沟通式去幻觉） | "先问再编造" | ChatDev 模式：执行者在细节缺失时询问规划者，而非自己编造。 |
| Critic（批评者） | "LLM 审阅者" | 主观的、有主见的审阅者。捕捉品味问题。可能被看似合理的文字迷惑。 |
| Verifier（验证者） | "确定性检查" | 基于代码的通过/失败。测试运行器、类型检查器、Schema 验证器。不会被迷惑。 |
| Verification gap（验证缺口） | "没人检查" | 21.3% 的 MAST 失败。答案在没有本可以捕获 bug 的检查的情况下发布。 |
| Revision loop（修订循环） | "批评者打回重做" | 批评者拒绝触发执行者带反馈的重新运行。需要预算。 |
| All-LLM anti-pattern（全 LLM 反模式） | "看起来不错" | 每个角色都是 LLM，没有确定性检查。经典 MAST 失败。 |
