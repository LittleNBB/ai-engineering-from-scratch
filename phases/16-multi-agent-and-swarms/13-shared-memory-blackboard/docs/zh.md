# 共享记忆与黑板模式（Shared Memory and Blackboard Patterns）

> 2026 年多智能体系统中共存两种方法：**消息池**（每个人看到每个人的消息，如 AutoGen GroupChat 或 MetaGPT）和**带订阅的黑板**（智能体订阅相关事件，如 Context-Aware MCP 或 Matrix 框架）。两者都是多智能体系统中唯一有状态的部分——这意味着有趣的 bug 都在这里。参考失败模式是**记忆污染（memory poisoning）**：一个智能体编造了一个"事实"，其他智能体将其视为已验证，准确率以比立即崩溃更难调试的方式逐渐衰减。本课用标准库构建两种结构，注入污染攻击，并展示在生产中真正有效的三种缓解措施。

**类型：** Learn + Build（学习 + 构建）
**语言：** Python（标准库，`threading`）
**前置课程：** Phase 16 · 04（Primitive Model），Phase 16 · 09（Parallel Swarm Networks）
**时间：** ~75 分钟

## 问题

多智能体系统需要一个地方让智能体共享事实。一个字面选项是"在消息中传递一切"——但那是在用额外复制重新发明共享状态。另一个是"给每个人一个全局日志"——但全局日志无限增长且容易被污染。第三个是"为每个智能体投影一个视图"——可扩展但 Schema 重。

当其中一个智能体产生幻觉并将幻觉写入共享状态时，每个读取该状态的下游智能体都会将幻觉当作事实采纳。等人类注意到时，推理链已经深入五步，而根本原因是写入的第三条消息。调试多智能体准确率衰减比调试崩溃更难。

这就是记忆污染。它是 MAST 分类法（Cemri 等，arXiv:2503.13657）中第二多记录的失败族，而且它是结构性的：任何没有溯源和不可写验证器的共享记忆设计最终都会表现出这种问题。

## 核心概念

### 两种主要拓扑

**全消息池。** 每个智能体读取每条消息。AutoGen GroupChat 和 MetaGPT 使用这种模式。简单、透明、可检查，但不能扩展到约 10 个智能体以上，因为每个智能体的上下文被其他智能体的工作填满。

```
智能体-A ──写入──▶ ┌────────────────┐ ◀──读取── 智能体-D
                    │ 消息池         │
智能体-B ──写入──▶ │                │ ◀──读取── 智能体-E
                    │ （全局日志）    │
智能体-C ──写入──▶ └────────────────┘ ◀──读取── 智能体-F
```

**带订阅的黑板。** 智能体声明对主题的兴趣；底层只路由相关消息。CA-MCP（arXiv:2601.11595）和 Matrix 去中心化框架（arXiv:2511.21686）使用这种模式。可扩展更远，但需要预先的 Schema 设计使订阅有意义。

```
                   ┌─ 主题: prices ──┐
智能体-A ──发布──▶ │                  │ ──▶ 智能体-D（已订阅）
                   ├─ 主题: orders ──┤
智能体-B ──发布──▶ │                  │ ──▶ 智能体-E（已订阅）
                   ├─ 主题: alerts ──┤
智能体-C ──发布──▶ │                  │ ──▶ 智能体-F（已订阅）
                   └──────────────────┘
```

### 各自适用场景

- **全池**在智能体少（< 10）、异质、对话短视界时胜出。当所有人看到一切时，推理谁说了什么是微不足道的。
- **黑板**在智能体多、角色同质但实例众多（群体）、对话长时间运行时胜出。路由节省 token 成本和上下文污染。

生产系统通常混合使用：顶层一个小全池（规划层），下方是黑板（工人层）。

### 记忆污染的一个场景

三个智能体处理一个研究任务。智能体 A 是检索智能体。智能体 B 是摘要智能体。智能体 C 是分析智能体。

1. A 获取一个页面并写入共享状态："研究报告显示 42% 的准确率提升。"
2. 获取的页面实际说的是"4.2% 的提升"。A 编造了一个小数点。
3. B 读取共享状态，写入："报告了巨大的 42% 准确率提升（来源：A）。"
4. C 读取共享状态，写入："建议采用——42% 的提升是变革性的。"
5. 最终报告引用了一个从未存在的 42% 数字。

没有智能体崩溃。没有测试失败。系统"正常工作"。幻觉通过共享状态从一个智能体的上下文跨越到每个下游智能体的推理中。

### 为什么这是结构性的

没有共享状态，智能体 A 的幻觉停留在 A 的上下文中。下游智能体会重新获取或重新推导，可能捕获错误。有了朴素的共享状态，A 的上下文变成了每个人的上下文，幻觉被洗白成了事实。

问题不是共享状态本身——而是**没有溯源且没有独立验证器**的共享状态。三种缓解措施可以解决：

1. **每次写入记录溯源。** 共享状态中的每个条目记录谁写的、什么时候、在什么提示词下、以及（如果适用）智能体引用了什么来源。下游智能体带着与溯源相关的怀疑态度阅读。
2. **版本化写入；视为仅追加。** 纠正是一个取代旧条目的新条目，而非就地更新。审计追踪被保留。
3. **至少保持一个不能写入共享状态的智能体。** 一个只读的验证者智能体采样条目、重新获取来源并标记不一致。因为它不能写入池中，所以不会被池污染。

### 黑板先例（Hayes-Roth，1985）

黑板模式比 LLM 智能体早四十年。Hayes-Roth（1985，"A Blackboard Architecture for Control"）描述了观察全局黑板、贡献部分解决方案并触发其他知识源的专业知识源。2026 年的黑板（CA-MCP、Matrix）是相同的模式，只是 LLM 智能体作为知识源，JSON 块作为部分解决方案。旧文献已记录了写入争用、机会主义控制和一致性的解决方案，现代系统在重新发现这些。

### 投影 vs 全视图

纯黑板给每个订阅者相同的投影（主题范围的）。更激进的设计是**每个智能体的投影**：每个智能体获得为其角色定制的视图。LangGraph 的状态 reducer 是 2026 年的经典实现——reducer 函数将全局状态折叠为角色特定的切片。

每个智能体的投影可扩展更远，但需要 Schema。没有 Schema，你会在每个智能体的提示词中重建临时投影。

### 写入争用模式

多个智能体同时写入是一个并发问题，不仅仅是 LLM 问题。三种模式有效：

- **串行写入者（单生产者）。** 所有写入通过一个协调者智能体进行序列化。简单，但是瓶颈。
- **带版本的乐观并发。** 每个条目有版本号；版本不匹配时写入失败并重试。经典数据库技术。
- **主题分区。** 不同智能体拥有不同主题。没有跨主题争用。需要设计分区边界。

大多数 2026 年框架默认使用串行写入者，因为 LLM 调用足够慢，争用罕见且瓶颈不会造成伤害。

### 不可写验证器

最有承重能力的缓解措施是只读验证器。实现规则：

- 验证器与团队共享状态（读取黑板或池）。
- 验证器对共享状态没有写入句柄——只能写入单独的验证通道。
- 验证器独立获取写入中引用的来源。标记分歧。
- 验证器自己的输出被路由到人类或单独的决策智能体，永远不反馈到池中。

没有这种分离，验证器的输出就变成池中的新条目，意味着被污染的池污染验证器，验证器又污染其验证。

## 动手实现

`code/main.py` 用标准库 Python 实现了两种拓扑加上一个玩具污染攻击和三种缓解措施。

- `MessagePool` — 线程安全的仅追加日志，完全读出。
- `Blackboard` — 按主题键的发布/订阅，每个智能体有订阅。
- `ProvenanceEntry` — 每次写入记录（写入者、时间戳、prompt_hash、source_uri）。
- `PoisoningScenario` — 运行一个三智能体研究任务，智能体 A 编造了一个小数。打印最终报告。
- `Verifier` — 一个只读智能体，重新获取来源并标记不一致。在有验证器的情况下运行相同场景。

运行：

```
python3 code/main.py
```

预期输出：
- 运行 1（无验证器）：编造的 42% 传播到最终报告。
- 运行 2（有验证器）：验证器标记不一致，池被标记为"已标记"，最终报告包含撤回。

## 实际应用

`outputs/skill-memory-auditor.md` 是一个技能，审计任何多智能体系统的共享记忆设计的溯源、版本控制和验证器分离。在新多智能体架构投入生产之前运行它。

## 交付建议

对于任何共享记忆设计：

- 每次写入记录溯源：`(writer, timestamp, prompt_hash, tool_calls_cited, source_uri)`。
- 使日志仅追加。纠正是引用被取代条目的新条目。
- 部署至少一个具有独立来源访问权的只读验证者智能体。
- 将验证者输出路由到单独的通道，而非回流到共享池。
- 记录被取代写入的比例——比例上升是幻觉模式的早期证据。

## 练习

1. 运行 `code/main.py`。确认运行 1 传播幻觉，运行 2 捕获它。
2. 添加第二个幻觉：智能体 B 编造了一个数据集大小。验证者应该能在不针对任何一个进行手动调优的情况下捕获两者。
3. 将全池切换为带主题分区的黑板（`prices`、`summaries`、`analyses`）。主题分区使哪些污染场景更难得逞，对哪些没有帮助？
4. 阅读 Hayes-Roth（1985，"A Blackboard Architecture for Control"）。识别本文未讨论但 2026 年系统会受益的论文中的两种控制模式。
5. 阅读 CA-MCP（arXiv:2601.11595）。将其 Shared Context Store 映射到 `code/main.py` 中的 MessagePool 或 Blackboard 类。CA-MCP 在其上添加了哪些原语？

## 关键术语

| 术语 | 常见说法 | 实际含义 |
|------|---------|---------|
| Message Pool（消息池） | "共享聊天历史" | 每个智能体读取的仅追加日志。完全透明，扩展性差。 |
| Blackboard（黑板） | "共享工作区" | 按主题键的发布/订阅。智能体订阅相关主题。可扩展更远。 |
| Provenance（溯源） | "谁写了什么" | 每次写入的元数据：写入者、时间戳、提示词、来源。 |
| Memory poisoning（记忆污染） | "幻觉传播" | 一个智能体的错误进入共享状态，下游智能体将其当作事实采纳。 |
| Append-only（仅追加） | "无就地更新" | 纠正是取代的新条目。保留审计追踪。 |
| Unwritable verifier（不可写验证器） | "独立审计者" | 重新获取来源并标记不一致的只读智能体。 |
| Projection（投影） | "范围视图" | 从全局状态计算的每个智能体视图。LangGraph reducer 是经典案例。 |
| Knowledge Source（知识源） | "专家智能体" | Hayes-Roth 1985 年对黑板参与者的术语。 |
