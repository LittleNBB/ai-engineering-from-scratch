# 监督者/编排者-工人模式（Supervisor / Orchestrator-Worker Pattern）

> 一个主导智能体负责规划和委派；专业化工人在并行上下文中执行并汇报。这是 Anthropic 研究系统背后的模式（Claude Opus 4 作为主导，Sonnet 4 作为子智能体），在内部研究评估中比单智能体 Opus 4 提升了 +90.2%。Anthropic 的工程博文报告称，BrowseComp 上 80% 的方差仅由 token 使用量解释——多智能体的优势主要来自每个子智能体获得全新的上下文窗口。本课从原语构建监督者模式，并涵盖 2026 年生产部署中的工程经验。

**类型：** Learn + Build（学习 + 构建）
**语言：** Python（标准库，`threading`）
**前置课程：** Phase 16 · 04（Primitive Model）
**时间：** ~75 分钟

## 问题

研究是单智能体系统失败的典型任务。你问"2023 到 2026 年间多智能体系统有什么变化？"单个智能体按顺序读五篇论文，用文本填满了一半上下文，然后不得不对所有论文一起推理。读到第五篇时已经忘了第一篇。它无法并行化。

监督者模式解决了这个问题：一个主导智能体规划搜索，将每个子问题委派给一个工人，然后综合。每个工人获得自己的 20 万 token 窗口来处理一个窄问题。主导智能体从未看到原始论文——只看到工人的摘要。

Anthropic 的生产研究系统报告在内部研究评估中比单个 Opus 4 提升了 +90.2%。同一篇博文指出，BrowseComp 上 80% 的方差仅由*token 使用量*解释。每个子智能体拥有全新上下文是主要机制。

## 核心概念

### 模式

```
                 ┌──────────────┐
                 │   主导       │  规划、分解、
                 │  (Opus 4)    │  综合
                 └──┬────┬───┬──┘
                    │    │   │
            ┌───────┘    │   └───────┐
            ▼            ▼           ▼
      ┌─────────┐  ┌─────────┐  ┌─────────┐
      │ 工人 1  │  │ 工人 2  │  │ 工人 3  │
      │(Sonnet) │  │(Sonnet) │  │(Sonnet) │
      └─────────┘  └─────────┘  └─────────┘
         全新        全新         全新
         上下文      上下文       上下文
```

主导智能体从不阅读原始材料。工人们在主导综合之前看不到彼此的工作。每条箭头都是一次携带窄工件的移交。

### 为什么它能赢

三个机制：

1. **每个子智能体拥有全新上下文。** 探索"FIPA-ACL 遗产"的工人不会携带主导规划时花费的 4 万 token。它为一个问题获得 20 万窗口。
2. **通过提示词专业化。** 主导的提示词是"分解和综合"，而非"研究"。每个工人的提示词是窄的："找出 X 中的变化"。专注的提示词产出专注的输出。
3. **并行。** 工人们并发运行。挂钟时间大约是 `max(各工人时间) + 规划 + 综合`，而非 `sum(各工人时间)`。

### 工程经验（Anthropic 2025）

Anthropic 的博文列出了几个在 2026 年仍然相关的生产经验：

- **按查询复杂度调整投入。** 简单查询：一个智能体，3-10 次工具调用。复杂查询：10+ 个智能体。主导智能体必须估算这一点，而非调用者。
- **先广后窄。** 先将问题分解为宽泛的子问题，如果答案需要深度，再为每个子问题生成更多工人。
- **彩虹部署（Rainbow deployments）。** 智能体是长时间运行且有状态的。传统的蓝绿部署不适用。Anthropic 使用彩虹：新版本渐进式推出，同时旧版本慢慢排空。
- **Token 使用量占主导。** 多智能体的 token 消耗约为单智能体的 15 倍。只在任务价值证明成本合理时才运行。

### LangGraph 的转变

LangGraph 最初发布了一个 `langgraph-supervisor` 库，带有高级的 `create_supervisor` 辅助函数。2025 年，LangChain 将建议改为通过直接使用工具调用来实现监督者模式，因为工具调用对*监督者看到什么*有更多控制（上下文工程）。该库仍然可用；文档现在推荐工具调用形式。

### 失败模式

- **主导编造计划。** 如果主导生成的子问题没有正确分解真实问题，工人们会在错误目标上进行精确研究。
- **工人过度探索。** 没有明确的范围边界，工人会超出指定子问题并污染综合步骤。
- **综合冲突。** 两个工人返回矛盾的事实。主导要么重新询问（增加一轮），要么明确记录分歧。静默选择一方是最糟糕的失败：用户永远不知道发生了分歧。

### 何时不该用监督者

- **串行任务。** 如果步骤 2 字面上需要步骤 1 的输出，并行化毫无益处。使用流水线（CrewAI Sequential、LangGraph 线性图）。
- **简单查询。** 单智能体更快更便宜。在生成工人之前使用主导的"调整投入"检查。
- **严格确定性。** 监督者使用 LLM 选择的委派。当审计/重放比适应性更重要时，静态图更好。

## 动手实现

`code/main.py` 使用 `threading` 实现了一个监督者管理三个并行工人的系统。主导将查询分解为子问题，工人们在每个子问题上并发运行，主导进行综合。没有真正的 LLM——工人们被脚本化以模拟获取-摘要。

关键结构：

- `Lead.plan(query)` 将查询拆分为 3 个子问题。
- `Worker.run(sub_q)` 返回一个假摘要（在生产中可以是任何使用工具的智能体）。
- `Lead.run(query)` 在线程中启动工人们，join，然后综合。

运行：

```
python3 code/main.py
```

输出展示了计划、带有开始/结束时间戳的并行工人跟踪，以及最终综合。你可以看到挂钟收益：三个 0.3 秒的工人在约 0.35 秒内运行完，而非 0.9 秒。

## 实际应用

`outputs/skill-supervisor-designer.md` 接受用户查询并产出监督者模式设计：主导系统提示词、工人角色、子问题分解规则和综合模板。在构建新的研究型智能体系统之前使用它。

## 交付建议

部署监督者模式前的检查清单：

- **模型配对。** 主导使用推理级模型（Opus 级、`o3` 级）。工人使用更快、更便宜的模型（Sonnet、`o4-mini`）。
- **工人超时。** 超过中位运行时间 2 倍的任何工人都会被终止；主导要么以更窄的范围重新生成，要么在没有它的情况下继续。
- **每个工人的 token 上限。** 硬性限制（比如预期综合输入的 10 倍）防止失控的工人耗尽预算。
- **可观察性。** 追踪主导的计划、每个工人的工具调用和综合。这是事后调试的基础。
- **彩虹推出。** 有状态的长时间运行智能体需要渐进式版本过渡，而非热切换。

## 练习

1. 运行 `code/main.py`，然后修改主导以生成 5 个工人而非 3 个。观察挂钟效果。在这个演示中，工人数达到多少时生成开销会超过并行收益？
2. 实现工人超时：终止运行超过 0.5 秒的任何工人，让主导综合剩余结果。你需要什么可观察性来知道工人被截断了？
3. 为主导的综合添加冲突检测步骤：如果两个工人返回矛盾的答案，主导记录分歧而非选择一个。你如何在不调用 LLM 的情况下检测矛盾？
4. 阅读 Anthropic 的研究系统工程博文。列出这个玩具演示在生产中运行需要采用的三个实践。
5. 比较 LangGraph 的 `create_supervisor`（旧版）与新的工具调用建议。哪个对监督者看到什么有更好的控制？为什么 Anthropic 明确只将子答案而非原始工人上下文传入综合？

## 关键术语

| 术语 | 常见说法 | 实际含义 |
|------|---------|---------|
| Supervisor（监督者） | "主导智能体" | 负责规划、委派和综合的编排器智能体。不亲自执行工作。 |
| Worker（工人） | "子智能体" | 由监督者调用的专注智能体，具有窄范围和自己的上下文窗口。 |
| Orchestrator-worker（编排者-工人） | "监督者模式" | 同一事物，不同名称。2026 年文献两者都用。 |
| Fresh context（全新上下文） | "干净窗口" | 工人的上下文从其系统提示词和分配的问题开始，而非主导的历史。 |
| Rainbow deployment（彩虹部署） | "渐进推出" | 长时间运行的有状态智能体需要版本化的排空-替换，而非蓝绿部署。 |
| Token dominance（Token 主导） | "上下文是变量" | 根据 Anthropic，研究评估中 80% 的方差来自总 token 使用量，而非模型选择。 |
| Scale effort（调整投入） | "匹配智能体数量与复杂度" | 主导估算查询难度，相应地生成 1 个 vs 10+ 个工人。 |
| Synthesis conflict（综合冲突） | "工人们意见不一" | 两个工人返回矛盾的事实；主导必须暴露分歧，而非静默选择一个。 |
