# 分层架构及其失败模式（Hierarchical Architecture and Its Failure Mode）

> 分层是嵌套的监督者。管理者智能体管理子管理者，子管理者管理工人。CrewAI 的 `Process.hierarchical` 是教科书版本：一个 `manager_llm` 动态委派任务并验证输出。LangGraph 的等价物是 `create_supervisor(create_supervisor(...))`。当任务是真实的组织架构图时，这是自然的模式。但它也是最有可能坍缩为管理循环的模式——管理者智能体分配工作不当、误解子输出或无法达成共识。串行流水线往往能打败它。

**类型：** Learn + Build（学习 + 构建）
**语言：** Python（标准库）
**前置课程：** Phase 16 · 05（Supervisor Pattern）
**时间：** ~60 分钟

## 问题

一旦监督者模式奏效，自然的下一步是"如果工人本身也是监督者呢？"团队有子团队；公司有部门的部门。分层架构映射了这一点。

问题是：LLM 管理者与人类管理者不同。人类管理者对其下属知道什么有稳定的先验认知。LLM 管理者每轮都根据上下文中的内容重新推理组织结构。上下文中微小的漂移，整个树就会错误分配工作。

## 核心概念

### 形态

```
                 管理者
                 ┌─────┐
                 └──┬──┘
           ┌────────┴────────┐
           ▼                 ▼
       子管理者 A         子管理者 B
       ┌─────┐           ┌─────┐
       └──┬──┘           └──┬──┘
         ┌┴──┬──┐          ┌┴──┐
         ▼   ▼  ▼          ▼   ▼
       W1  W2  W3         W4  W5
```

每个内部节点负责规划、委派和综合。只有叶子节点执行工作。

### 适用场景

- **清晰的组织映射。** 如果真实任务是部门化的（"法务审文档、财务审文档、工程审文档，然后给高管汇总"），层级结构是显式的。
- **局部综合。** 每个子管理者在顶层管理者看到之前综合其团队的输出。顶层管理者看到三个子管理者摘要，而非十五个工人输出。

### 失败模式

2026 年的事后分析不断发现三种失败模式：

1. **任务分配错误。** 管理者读取目标，编造一种分解方式，并委派给错误的子管理者。因为子管理者顺从地处理被分配的工作，错误只在顶层综合时才浮现——距离人类本可以捕获它的地方隔了一层。
2. **输出误解。** 子管理者返回"无法验证声明 X"。顶层管理者汇总为"声明 X 未确认"。含义在每一层都会漂移。
3. **共识循环。** 两个子管理者意见不一；顶层管理者要求它们调和；它们重新向下委派；工人重新运行；子管理者返回略有不同的答案；循环。CrewAI 的 `Process.hierarchical` 用步骤限制来防护，但限制本身现在成了超参数。

### 决定性问题

串行（线性流水线）vs 分层：你的任务是否真的有独立的子团队，还是一个伪装成树的线性流？如果是后者，用串行。如果是前者，用分层，但要预算显式的调和规则。

### CrewAI 的实现

`Process.hierarchical` 在专家团队上方接线一个管理者 LLM。管理者：

- 接收顶层任务，
- 将子任务分配给团队，
- 评估团队输出，
- 决定是接受、重新委派还是迭代。

文档：https://docs.crewai.com/en/introduction（在 Core Concepts 下查找 "Hierarchical Process"）。

### LangGraph 的实现

LangGraph 使用嵌套的 `create_supervisor` 调用。内部监督者有自己的图；外部监督者将内部图视为不透明节点。这对调试来说比 CrewAI 更干净（你可以分别单步执行每个图），但更难表达树的动态重塑。

参考：https://reference.langchain.com/python/langgraph-supervisor。

## 动手实现

`code/main.py` 运行一个 3 层级结构：

- 顶层管理者：将任务拆分为"工程"和"法务"分支，
- 工程子管理者：拆分为"前端"和"后端"工人，
- 法务子管理者：一个工人。

演示对比了正常路径（所有人一致）和**扰动路径**——顶层管理者的分解将"法务"误标为"财务"，观察错误如何级联：子管理者顺从地做财务工作，顶层综合者报告财务发现，原始的法务问题无人回答。

运行：

```
python3 code/main.py
```

输出展示了两条路径，清晰并排对比"问了什么"和"交付了什么"。

## 实际应用

`outputs/skill-hierarchy-fitness.md` 评估给定任务应使用分层、串行还是扁平监督者。输入：任务描述、组织结构、调和预算。输出：模式推荐及需要防范的特定失败模式。

## 交付建议

如果你要交付分层架构：

- **将树深度限制在 2。** 三层已经会让大多数错误脱离可观察性。
- **显式调和预算。** 设置顶层管理者必须提交前的最大轮次。通常是 2。
- **每次综合附带溯源。** 每个节点的摘要必须引用哪些叶子输出产出了它。
- **对分解漂移发出告警。** 记录管理者每步的分解；与用户查询对比。如果分解不再覆盖查询，触发告警。

## 练习

1. 运行 `code/main.py` 并对比正常路径与扰动路径。在顶层输出与用户问题完全偏离之前，需要经过多少层管理者移交？
2. 添加第三层（顶层 → 子层 → 子子层 → 工人）。随着深度增长，测量扰动路径自我纠正与完全偏离的频率。
3. 在每个子管理者处实现一个"金丝雀"工人，总是被问及原始用户问题（不做修改）。用金丝雀答案检测分解漂移。当金丝雀与综合答案不一致时，管理者应如何反应？
4. 阅读 CrewAI 的 `Process.hierarchical` 文档。识别 CrewAI 应用的一个具体防护措施（步骤限制、manager_llm 约束），并描述它针对什么失败模式。
5. 比较嵌套 LangGraph 监督者与 CrewAI 分层。哪个使调和循环更容易检测？

## 关键术语

| 术语 | 常见说法 | 实际含义 |
|------|---------|---------|
| Hierarchical（分层） | "组织架构图模式" | 监督者之上的监督者；只有叶子执行工作。 |
| Manager LLM（管理者 LLM） | "老板" | 在内部节点进行分解、分配和验证的 LLM。 |
| Decomposition drift（分解漂移） | "老板迷失了方向" | 顶层管理者的拆分不再覆盖原始问题。 |
| Reconciliation loop（调和循环） | "无尽的会议" | 子管理者意见不一；顶层重新委派；工人重新运行；循环直到预算耗尽。 |
| Depth-2 ceiling（深度-2 上限） | "不要超过 2 层" | 经验性防护：3+ 层会使可观察性坍缩。 |
| Canary question（金丝雀问题） | "每层的基准真相" | 一个总是被问及原始查询（不做修改）的工人，用于检测漂移。 |
| Provenance chain（溯源链） | "谁说了什么" | 从每个综合回溯到产出它的叶子输出的追踪。 |
