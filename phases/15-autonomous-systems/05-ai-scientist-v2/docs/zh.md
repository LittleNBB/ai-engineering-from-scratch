# AI Scientist v2 — 工作坊级别的自主研究（Workshop-Level Autonomous Research）

> Sakana 的 AI Scientist v2（Yamada 等，arXiv:2504.08066）运行完整的研究循环：假设、代码、实验、图表、撰写、投稿。它是第一个生成论文通过 ICLR 2025 工作坊同行评审的系统。独立评估（Beel 等）发现 42% 的实验因编码错误而失败，文献综述经常将已建立的概念误标为新颖。Sakana 自己的文档警告代码库执行 LLM 编写的代码并建议 Docker 隔离。这幅图景的两面都是重点。

**类型：** Learn（学习）
**语言：** Python（标准库，研究循环状态机玩具）
**前置课程：** Phase 15 · 03（AlphaEvolve），Phase 15 · 04（DGM）
**时间：** ~60 分钟

## 问题

研究是一个开放式任务。与 AlphaEvolve 的算法搜索或 DGM 的基准约束自修改不同，研究结果没有机器可检查的正确性标准。论文由审稿人评判，而非单元测试。这使得循环更难闭合——如果闭合则更有价值，因为研究是复合进步所在的地方。

AI Scientist v1（Sakana，2024）通过从人类编写的模板开始来闭合循环。LLM 在固定脚手架内填充实验。AI Scientist v2（Yamada 等，2025）通过使用带有视觉语言模型批评循环的智能体树搜索来移除模板要求。系统生成想法、实现实验、产生图表、撰写论文，并根据审稿反馈迭代。

同行评审结论：一篇 v2 生成的论文在 ICLR 2025 工作坊被接受（有披露）。独立评估结论：该系统远未可靠。两者都是真的。

## 核心概念

### 架构

1. **想法生成。** LLM 根据主题和先前文献提出研究想法。v1 使用模板；v2 使用假设空间上的智能体搜索。
2. **新颖性检查。** 文献检索步骤检查想法是否已被发表。这是 Beel 等评估发现误标的步骤——已建立的方法经常被分类为新颖。
3. **实验计划。** 智能体起草实验方案并编写代码。
4. **执行。** 代码在沙箱中运行。失败被反馈到重试循环中。在 Beel 等的测量中，42% 的实验在此阶段因编码错误而失败。
5. **图表生成。** 视觉语言模型读取生成的图表并重写以提高清晰度。这是 v2 的关键技术添加。
6. **撰写。** LLM 起草论文，与内部审稿人迭代。
7. **可选：投稿。** 论文被提交到一个会议。

### 工作坊接受结果意味着什么

一篇 v2 生成的论文通过了 ICLR 2025 工作坊的同行评审。作者向程序委员会披露了论文的来源。接受是一个数据点；它不是声称系统"做研究"的许可证。

重要背景：工作坊论文的门槛低于主会议论文。同行评审是有噪声的；在任何给定日期只有一小部分投稿被接受。一次成功是概念证明，不是可靠性声明。Nature 2026 论文记录了端到端循环，本身由人类研究者共同撰写；它不是"系统写了一篇 Nature 论文"。

### 独立评估发现了什么

Beel 等（arXiv:2502.14297）运行了外部评估。主要发现：

- **实验失败。** 42% 的实验因编码错误（错误导入、形状不匹配、未定义变量）而失败。重试循环捕获了一些，但不是全部。
- **新颖性误标。** 文献检索步骤经常将已建立的概念标记为新颖。这是研究版的幻觉。
- **呈现质量差距。** 视觉语言图表批评产生了出版级的视觉效果，掩盖了底层的实验弱点。

最后一个发现对本阶段很重要。一个在没有进行令人信服的研究的情况下产生令人信服的输出的系统，比一个明显失败的系统更危险，而非更安全。评估必须触及底层声明，而非停留在图表层面。

### 沙箱逃逸担忧

Sakana 自己的仓库 README 警告：

> 由于此软件的性质，它执行 LLM 生成的代码，我们无法保证安全。存在危险包、不受控制的网络访问和生成意外进程的风险。使用风险自负并考虑 Docker 隔离。

这是未验证领域中自主性的操作形态。LLM 编写代码；代码运行；代码可以做进程被允许做的任何事情。没有对文件系统、网络和进程操作进行硬限制的沙箱，任何自导向的研究智能体都可以窃取数据、消耗计算或重写自己。

AlphaEvolve 的沙箱故事更容易，因为它的评估器很紧。AI Scientist v2 的循环运行开放式目标的开放式代码。这就是为什么它需要更强的隔离（最低 Docker；首选 seccomp / gVisor）以及在每个投稿离开系统之前进行人工审查。

### v2 在前沿技术栈中的位置

| 系统 | 目标 | 输出类型 | 评估器 | 已知失败 |
|---|---|---|---|---|
| AlphaEvolve | 算法 | 代码 | 单元测试 + 基准 | 受限于评估器严谨性 |
| DGM | 智能体脚手架 | 代码 | SWE-bench | 奖励黑客 |
| AI Scientist v2 | 研究论文 | 文本 + 代码 + 图表 | 同行评审（弱） | 实验失败、误标、打磨掩盖弱点 |

v2 在三者中拥有最弱的自动评估器、最宽的输出表面和最短的公共工件路径。操作控制（沙箱、审查、披露）承担了大部分安全工作。

## 动手实现

`code/main.py` 将 v2 循环模拟为状态机：想法 → 新颖性检查 → 实验 → 图表 → 撰写 → 审查 → 接受或迭代。每个状态有来自 Beel 等发现的可配置失败概率。运行模拟器 N 次循环并计数：

- 多少想法到达投稿。
- 多少投稿会有精心打磨的论文掩盖的关键实验缺陷。
- 重试预算如何在质量和产出之间权衡。

## 实际应用

`outputs/skill-ai-scientist-sandbox-review.md` 是研究循环智能体在任何产物离开沙箱之前的两道门审查检查清单。

## 练习

1. 使用默认参数运行 `code/main.py`。多少比例的循环运行产生"干净"的论文？多少比例产生了图表批评打磨过的有实验失败缺陷的论文？

2. 默认已使用 Beel 等的 42% / 25%。用 `--experiment-failure 0.20 --novelty-mislabel 0.10` 然后用 `--experiment-failure 0.60 --novelty-mislabel 0.40` 重新运行。打磨过但有缺陷的比例在两次运行之间如何变化？

3. 阅读 Sakana 的 AI Scientist v2 仓库 README 关于沙箱要求。说出你会为多天自主运行应用的两个额外限制（除 Docker 外）。

4. 阅读 Beel 等关于呈现质量差距的第 4 节。设计一个能捕获看起来精美但实验有缺陷的论文的额外评估器。

5. 为研究智能体输出提出一个比"博士阅读每篇论文"更好扩展的人工审查协议。识别瓶颈并围绕它设计。

## 关键术语

| 术语 | 常见说法 | 实际含义 |
|---|---|---|
| AI Scientist v1 | "Sakana 的模板化研究智能体" | 在固定脚手架中填充实验 |
| AI Scientist v2 | "无模板研究智能体" | 带 VLM 图表批评的智能体树搜索 |
| Agentic tree search（智能体树搜索） | "分支研究智能体" | 并行展开多个实验计划；由内部批评者修剪 |
| Vision-language critique（视觉语言批评） | "VLM 图表打磨" | 多模态模型读取图表并重写以提高清晰度 |
| Literature retrieval（文献检索） | "新颖性检查" | 搜索先前工作以确认想法新颖——有据可查的误标 |
| Polish masking（打磨掩盖） | "漂亮的论文，破碎的研究" | 呈现质量超过实验质量；掩盖弱点 |
| Sandbox escape（沙箱逃逸） | "LLM 代码突破" | 智能体执行的代码做了循环设计者没有意图的事情 |
