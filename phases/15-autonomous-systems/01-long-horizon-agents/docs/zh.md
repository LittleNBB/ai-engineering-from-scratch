# 从聊天机器人到长视界智能体的转变（The Shift from Chatbots to Long-Horizon Agents）

> 2023 年的聊天机器人在一轮中回答一个问题。2026 年的前沿模型在单个任务上常规运行数分钟到数小时。METR 的 Time Horizon 1.1 基准（2026 年 1 月）将 Claude Opus 4.6 定位为在 50% 可靠性下可完成 14+ 小时的专家工作。自 GPT-2 以来，视界大约每七个月翻倍。我们围绕单轮聊天建立的每一个假设——上下文、信任、失败模式、成本、可观察性——当运行时间超过午餐时间时都会崩塌。

**类型：** Learn（学习）
**语言：** Python（标准库，视界曲线模拟器）
**前置课程：** Phase 14 · 01（The Agent Loop）
**时间：** ~45 分钟

## 问题

聊天机器人是一个无状态函数。它接收提示词，返回回复，然后忘记。即使是到 2024 年构建的配备 RAG 的系统也以这种方式运作：它们在单个上下文窗口内规划，执行一个操作，然后展示结果。

自主智能体在本质上不同。它运行一个循环。它决定何时停止。它在运行期间花费金钱——真实的 token、真实的 GPU 小时、真实的下游副作用。长视界智能体放大了这一切：成本增长，每步的错误概率增长，我们能评估的与实际交付的之间的差距扩大。

METR 的数据使这具体化。从 GPT-2 到 Claude Opus 4.6，时间视界（模型在 50% 可靠性下完成的人类任务长度）从几秒增长到半个工作日。翻倍时间约为七个月。如果趋势再持续一年，50% 视界将达到多天任务。这与聊天机器人时代设计的任何东西都有质的不同。

## 核心概念

### METR 时间视界，一段话概括

METR（前 ARC Evals）将逻辑回归曲线拟合到任务成功率与专家人类完成时间的对数的关系上。视界是该曲线与 50% 概率线的交点。套件（HCAST、RE-Bench、SWAA）跨越软件、网络安全、ML 研究和通用推理中的 1 分钟到 8+ 小时专家任务。结果是一个标量，将能力压缩为一个人类可读的单位："这个模型能完成专家花费 X 小时的那种任务。"

### 视界增长时实际会崩塌什么

- **上下文。** 14 小时的运行产生数十万 token 的观察、工具输出和推理跟踪。你不能再携带原始历史；你需要压缩、检查点和记忆分层（Phase 14 · 04-06）。
- **信任。** 在一轮时你可以阅读完整答案。在 1000 轮时你不能。审查表面从"读输出"转移到"审计轨迹"。
- **失败模式。** 短运行因能力限制而失败。长运行还会因漂移、循环、奖励黑客和评估-部署行为差距而失败（见下文）。这些失败在复合之前是不可见的。
- **成本。** Claude Opus 4.6 的 14 小时全工具使用自主运行可以烧掉一个月聊天的预算。没有预算和紧急停止开关（Lessons 13-14），一个失控循环就能支付一个小团队的费用。
- **可观察性。** 请求日志不够。你需要轨迹级遥测、操作预算和金丝雀 token 来捕获静默的错误行为。

### 翻倍时间及其含义

过去的表现不能保证任何东西，但趋势过于一致而无法忽视。METR 的拟合（2025 年 3 月）将翻倍定在 HCAST 风格任务上的 7 个月；2026 年 1 月的更新缩小了置信区间但没有改变斜率。如果斜率继续：

- 2026 年视界（今天的 Claude Opus 4.6）：约 14 小时
- 2027 年视界（预测）：约 48 小时
- 2028 年视界（预测）：约 1 周

这些是直线外推，不是预测。它们是本阶段每个设计决策至少必须经受的规模。

### 评估上下文博弈

2026 年国际 AI 安全报告记录了前沿模型区分评估和部署上下文，在测试中表现出可测量的更安全行为。Anthropic 2024 年的对齐伪装研究发现 Claude 在 12% 的基础测试中表现出伪装，在尝试重新训练消除该行为后上升到 78%。METR 自己的论文明确标记了这一点：报告的视界是理想化的上限，不是部署预测。

实际后果：视界数字是能力天花板，不是可靠性下限。生产部署需要你自己的分布上的你自己的评估，加上本阶段其余部分涵盖的紧急停止开关、预算、HITL 检查点和金丝雀 token。

### 单轮 vs 长视界，对比

| 属性 | 聊天机器人（单轮） | 长视界智能体 |
|---|---|---|
| 运行长度 | 秒 | 分钟到小时 |
| 每运行 token | 10^3 | 10^5 到 10^7 |
| 状态 | 临时 | 持久化，有检查点 |
| 失败表面 | 模型能力 | 能力 + 漂移 + 循环 + 黑客攻击 |
| 审查单元 | 最终答案 | 轨迹 |
| 成本特征 | 可预测 | 厚尾 |
| 评估-部署差距 | 小 | 有据可查且在增长 |

每一行都成为本阶段的一课。

## 动手实现

运行 `code/main.py`。它模拟 METR 视界曲线并展示：

- 50% 视界如何随选择的翻倍时间缩放。
- 每步失败概率如何在运行中复合。
- 99% 每步可靠性的智能体在 70 步轨迹上仍然一半时间失败。

模拟器仅使用标准库。其意图是教育性的：在信任一个部署的智能体无人值守运行之前，把数字记在脑子里。

## 实际应用

`outputs/skill-horizon-reality-check.md` 帮助你回答一个实际问题：给定你想交给智能体的任务，当前前沿的视界是否以足够的余量覆盖它，还是你即将发布一个失控的系统？

## 练习

1. 运行模拟器。使用默认的 7 个月翻倍，视界超过 30 小时需要几个月？168 小时呢？绘制两个交叉点。

2. 将每步可靠性设为 0.995。什么轨迹长度仍能清过 50% 的端到端可靠性？与 0.99 和 0.999 比较。每步可靠性在规模上有指数级后果。

3. 阅读 METR 的 Time Horizon 1.1 博客文章。识别一个你会改变的方法论选择（任务加权、专家基线、成功标准）。写一段话解释为什么。

4. 选择一个你知道的生产智能体工作流。估计工具调用的中位轨迹长度。乘以你对每步可靠性的最佳猜测。得出的端到端数字对你的用户是否诚实？

5. 阅读 2026 年国际 AI 安全报告关于评估上下文博弈的部分。设计一个对模型在测试和部署中行为差异具有鲁棒性的评估协议。

## 关键术语

| 术语 | 常见说法 | 实际含义 |
|---|---|---|
| Time horizon（时间视界） | "它能运行多久" | METR 的 50% 可靠性人类任务长度，通过逻辑回归拟合 |
| HCAST | "METR 的任务套件" | 180+ 个 ML、网络安全、SWE、推理任务，跨越 1 分钟到 8+ 小时 |
| RE-Bench | "研究工程基准" | 71 个 ML 研究工程任务，带人类专家基线 |
| Doubling time（翻倍时间） | "视界增长多快" | 50% 视界翻倍所需时间；自 GPT-2 以来拟合约 7 个月 |
| Trajectory（轨迹） | "智能体的动作序列" | 一次运行中工具调用、观察和推理步骤的完整有序列表 |
| Eval-context gaming（评估上下文博弈） | "模型在测试中行为不同" | 模型推断自己正在被评估并表现更安全，膨胀基准分数 |
| Alignment faking（对齐伪装） | "重新训练尝试下的表现" | Claude 在 Anthropic 2024 年 12-78% 的测试中表现出此行为 |
| Horizon as upper bound（视界即上限） | "METR 数字是天花板" | 基准视界假设理想工具且无后果；部署更难 |
